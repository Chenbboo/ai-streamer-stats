# AI 主播直播数据统计后台 — 需求分析

版本:v0.1(2026-07-02)
状态:待确认

## 1. 背景

公司运营多名 TikTok(越南区)主播。每天的经营数据散落在:主播手工发的文字汇报(如 `Zhenzhen Ngày 1/7 Tổng 27079`)、直播打赏(Xu)排行榜截图、与客户的私信聊天截图。目前靠人工汇总,效率低、易出错、无法沉淀客户数据。

本系统基于 RuoYi-Vue3 底座,实现:截图/汇报上传 → AI 识别提取 → 人工校正 → 数据统计。

## 2. 用户角色

| 角色 | role_key | 权限 |
|------|----------|------|
| 主播 | streamer | 上传自己的截图和汇报;查看自己的统计 |
| 运营 | operator | 上传、校正全部数据;查看全部统计 |
| 直播管理员 | live_admin | 模块全部权限,含删除 |

老板查看汇总报表可用 live_admin 或后续单独加只读角色。

## 3. 数据源与提取字段

### 3.1 打赏排行截图(核心)

TikTok 直播观众送礼(Xu)排行榜,一场直播需多张截图滚动拍全。

AI 提取(**按客户明细**):

- 排名(如 13、14)
- 客户昵称(如 `JCFoto(제이씨포토)`、`_Pope_`,含 emoji/多语言)
- Xu 数(如 847、697)
- 徽章(如 MWRM 粉丝团标识,可选)

系统处理:同一场次多张截图**自动拼接去重**(按昵称合并,取排名连续性校验);识别出的客户自动沉淀到客户库,支持大客户、复购分析。

### 3.2 聊天记录截图

主播与客户的 TikTok 私信截图。统计口径:

- **互动客户数**:识别对话客户昵称,统计主播当天跟进了多少客户(工作量考核)
- **重点客户跟进记录**:截图与客户关联留存,可按客户查看历史跟进记录(证据留存)

不逐条识别聊天内容,只提取:客户昵称、截图时间(若可见)、消息大致条数(可选)。

### 3.3 文字工作汇报

格式如 `Zhenzhen Ngày 1/7 Tổng 27079`,即「主播名 + 日期 + 当日总 Xu」。AI 解析为结构化记录。

**注意:汇报总数与截图明细不做对账**,汇报数为主播自报口径,两者并列展示即可。

## 4. 业务流程

1. 主播(或运营代传)在「图片上传」页选择日期、类型(打赏榜/聊天/汇报文本),批量上传
2. 系统调大模型视觉 API 异步识别,结果存为「待校正」
3. 运营在「识别校正」页逐条核对 AI 结果,可修改昵称/Xu 数、合并重复客户,确认入库
4. 「数据统计」页按角色出报表

## 5. 统计需求

- **运营视角**:全部主播横向对比、日/周/月 Xu 趋势、主播排名、互动客户数排名
- **主播视角**:仅自己的每日 Xu、打赏客户明细、互动客户数趋势
- **老板视角**:周/月汇总,Top 主播、Top 客户,可导出 Excel
- 计量单位统一为 **Xu**,不做货币折算、不算提成

## 6. AI 识别方案

调用大模型视觉 API(如 Claude),输入截图 + 结构化提取 prompt,输出 JSON。要点:

- 需处理越南语/韩语/中文/emoji 混排昵称
- 识别置信度低的字段标记,校正页高亮
- 按调用计费,中等规模(每天几百张)成本可控;API key 配置在后端,不落前端

## 7. 数据模型草案

- `live_streamer` 主播(关联 sys_user)
- `live_customer` 客户(昵称、平台、首次/最近出现时间、累计 Xu)
- `live_upload` 上传记录(文件、类型、日期、主播、上传人、识别状态)
- `live_gift_record` 打赏明细(场次/日期、主播、客户、Xu、排名、来源截图、校正状态)
- `live_chat_record` 聊天跟进记录(日期、主播、客户、来源截图)
- `live_daily_report` 每日汇报(主播、日期、自报总 Xu、原文)

## 8. 边界(本期不做)

- 汇报与明细对账、差异告警
- Xu 折算货币 / 提成计算
- 聊天内容逐条识别与质检
- 直播平台 API 自动拉数(全部靠截图)

## 9. 已确认口径

1. **统计按「天」**,不引入场次概念;一天多场合并为当日数据
2. **客户身份以昵称为准**,接受改名导致的误差:
   - 识别时保存客户头像小图,新昵称出现时按头像相似度提示"疑似同一客户"(辅助)
   - 校正页提供**人工合并客户**功能:合并后历史打赏累计,系统保留昵称映射关系(兜底)

## 10. 待确认问题

1. 截图保留时长 / 存储上限
2. 主播上传是否有截止时间要求(如次日中午前传完前一天)

## 11. 表设计遗留问题(暂缓,做打赏识别前必须解决)

1. 打赏榜按场重置,采集需按场/批次区分,否则一天多场数据会被误去重
2. 昵称不唯一,`live_customer.uk_nickname` 需去掉,重名靠人工拆分
3. AI 识别结果应先暂存,人工确认后才创建客户,避免污染客户表

## 12. 国际化要求(中越双语)

系统支持中文和越南语两种语言,通过顶部导航栏的语言切换按钮切换。

### 12.1 翻译规范

**所有用户可见的文字必须使用 i18n 翻译**,包括但不限于:

- 页面标题、按钮文字、表单标签
- 表格列名、提示信息、错误消息
- 状态标签、下拉选项、占位符文字

### 12.2 开发规范

**每次代码改动必须同步处理翻译:**

1. **新增文字** → 同时添加中文(`zh-CN.js`)和越南语(`vi-VN.js`)翻译
2. **修改文字** → 同时更新两种语言的翻译
3. **新增页面** → 页面中的所有文字都使用 `$t('key')` 形式
4. **下拉选项** → 使用 `computed` 返回翻译后的选项列表

### 12.3 翻译文件位置

```
src/locales/
├── index.js      # i18n 配置
├── zh-CN.js      # 中文语言包
└── vi-VN.js      # 越南语语言包
```

### 12.4 菜单翻译

菜单名称存储在数据库 `sys_menu` 表中:
- `menu_name` — 中文名称
- `menu_name_vi` — 越南语名称

新增菜单时必须同时填写两种语言的名称。
